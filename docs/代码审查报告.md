## 一、P0 严重（影响财务数据正确性）

### A. 「非手动入口」的收付款不联动合同 paid_amount / 成本（根因级）⭐
手动收付款走 `paymentService.create`，会在事务内更新合同 `paid_amount` 并联动 `cost_record`。但**另外两个写入入口绕过了它，直接 `Payment.create`**，导致联动全部丢失：

1. **企微审批同步**：`backend/src/services/wechat/wechatSyncService.js` 的 `syncPayment`（约 296 行）创建 `category='business'`、`confirm_status='confirmed'` 的付款，但**未更新关联采购合同的 `paid_amount`**；`category='fee'` 时也**未写 `cost_record`**。
2. **历史数据导入**：`backend/src/services/importService.js` 的 `importPayment`（约 479 行）同样直接建记录、无联动。

**影响**：
- 企微/导入进来的采购付款，采购合同显示"未付款"，**应付虚高、合同进度失真**；
- 费用类付款不进 `cost_records`，**成本统计/看板成本饼图漏算**；
- 这正是为什么需要"冗余字段对账任务"，但根因应从写入端修复。

**建议**：抽出"联动逻辑"为可复用方法（更新合同 paid_amount + syncFromPayment），让 `syncPayment`、`importPayment` 在各自事务内调用；或这两个入口改为复用 `paymentService.create`。

---

### B. 库存「渠道成本」对空 `patent_type` 计算错误
`backend/src/services/inventoryService.js` `_attachChannelCosts`（约 1160-1173 行）模糊匹配：

```
if (patentType.includes(key) || key.includes(patentType)) { ... }
```

当 `patent_type` 为空字符串时，`key.includes('')` **恒为 true**，会命中遍历到的**第一个配置项**（如"发明"=1000），而不是 `default`（500）。

**影响**：未填专利类型的专利，`channel_cost` / `final_profit` 计算错误。
**建议**：模糊匹配前增加 `patentType` 非空判断（空则直接取 `default`）。

---

### C. 业绩重复上传导致提成翻倍
`backend/src/services/performanceUploadService.js` `confirmImport` **没有校验同 `year/month` 是否已存在 confirmed 批次**。若同月业绩被重复确认导入，`getMonthlyPerformance` 的 `SUM(performance_amount)` 会把多批次相加。

**影响**：销售提成基数翻倍 → **工资多发**。
**建议**：确认入库前检查同年月已有 confirmed 批次则拦截（或提示"覆盖/作废旧批次"）。

---

### D. 合同导入状态 `pending` 与模型枚举不一致 → 整批导入失败
`backend/src/services/importService.js` 的合同模板与 `validateRow` 允许 `status='pending'`，但 `Contract` 模型枚举是 `draft/active/completed/terminated`（**无 pending**）。

**影响**：用户按模板填 `pending`，校验通过但 `Contract.create` 触发数据库枚举错误，**整批回滚**导入失败。
**建议**：模板与校验改为 `draft`，或在导入时把 `pending` 映射为 `draft`。

## 二、P1 重要（功能 / 权限错误）

### E. 对账「手动匹配」可重复占用同一 payment
`backend/src/services/reconciliationService.js` `manualMatch`（约 358-381 行）仅校验账户与金额，**未校验目标 payment 是否已被其他流水匹配**。自动匹配 `_findMatch` 有排除逻辑，手动匹配没有。

**影响**：一个 payment 可被多条流水匹配，对账统计（matched/extra）失真。
**建议**：`manualMatch` 前查询该 `payment_id` 是否已被其他 `matched` 流水占用，是则拒绝。

### F. 前端 `isFinance` 判断了不存在的角色
`frontend/src/stores/user.js:19`：

```
const isFinance = computed(() => userInfo.value.role === 'finance' || userInfo.value.role === 'admin')
```

系统实际角色是 `admin/process/agent`，**没有 `finance`**。该判断对 `process`（财务/流程角色）恒为 false，等价于 `isAdmin`。

**影响**：若有用 `isFinance` 控制的财务功能/按钮，`process` 用户会被错误地隐藏。
**建议**：改为 `role === 'process' || role === 'admin'`（按业务确认 process 的定位）。

## 三、待业务确认（口径 / 语义）

### G. 代理专利「已售利润」口径不一致
`inventoryService.markAsSold` 写入的 `actual_profit = 成交价 - 采购价 - 维持成本`（毛利），**未按 `profit_rule` 扣除代理分成**；而在库列表的 `net_profit` 是分成后的。已售统计（`getSoldStats`/`getSoldAnalytics`）基于 `actual_profit`，会**高估代理专利的实际收益**。
**建议**：确认"已售利润"是否应为分成后口径；若是，`markAsSold` 用 `_calculateNetProfit` 调整。

### H. 销售提成「归属月 + 1 发放」语义
`payrollService.generate(year, month)` 用**同 year/month** 查 `performance_records`（归属月），但注释/设计写"归属月 + 1 工资条发放"。取决于 `Payroll.month` 的语义：
- 若 `month` = 工资"所属月"（=归属月），当前实现正确，"+1"指实际打款时间；
- 若 `month` = "发放月"，则应查**上一个月**归属的业绩，当前为 bug。
**建议**：明确 `Payroll.month` 定义后决定是否调整。

### I. 客户/供应商往来账「应收/应付」口径
`customerService/supplierService._getTransactionSummary` 的 `contract_total` 包含**所有状态**合同（含 `terminated`/`draft`），而看板 `dashboardService` 只算 `active/completed`。两处应收/应付口径不一致。
**建议**：统一为 `active/completed`（或明确各自用途）。

### J. 批量入库直填 `total_maintain_cost` 与对账任务冲突（重要提醒）
`inventoryBatchService` 允许 Excel 直接填 `total_maintain_cost`（历史成本），但**不创建对应 `patent_annual_fees` 明细**。而本次新增的"冗余字段对账任务"假设 `total_maintain_cost == SUM(annual_fees)`，会对这些专利**误报漂移**。
**建议**：二选一——(a) 对账任务移除 inventory 维度的校验；(b) 批量导入时按 `total_maintain_cost` 生成一条"期初维持成本"年费明细。

## 六、二次深入审查补充发现（定时任务 / Controller / 企微 / 告警）

### P1 新增：详情接口 `getDetail` 普遍未做数据隔离（越权读）⭐
列表接口对 `agent` 做了数据隔离，但**详情接口没有**：
- `paymentService.getDetail(id)`、`inventoryService.getDetail(id)`、`projectService.getDetail(id)`、`expenseService.getDetail(id)`、`loanService.getDetail(id)` 均**只按 id 查询、不校验归属**；对应 controller 也未传 `userId/userRole`。
- 仅 `contractService.getDetail(id, dataFilter)` 做了隔离。

**影响**：`agent` 角色可通过 `GET /payments/:id`、`/inventory/:id` 等**枚举 id 越权读取他人记录详情**（金额、利润、采购价、买家联系方式等敏感信息）。
**建议**：`getDetail` 增加归属校验（agent 时校验 `created_by`/`owner_id`，不符则 404/403）。

### 企微回调（P1/P2）
- **P1 验签可被绕过**：`wechatCryptoService.verifySignature` 在 `WECHAT_TOKEN` 未配置（空字符串）时仍照常计算 sha1，等于用空 token 验签，**攻击者可伪造回调触发同步**。建议：`token` 为空时直接判定失败。
- **P2 回调可能超时重试**：`wechatController.receiveCallback` 在返回前 `await wechatSyncService.handleCallback`（拉审批详情 + 写库），企微要求 5 秒内响应，链路慢时会超时→企微重试（虽 `sp_no` 幂等兜底）。建议：先 `send('success')`，再 `setImmediate` 异步处理。
- **P2 时序安全**：`verifySignature` 用 `===` 比较哈希，非时间安全比较（低风险）。

### 定时任务（P2）
- `contractReminderJob` 的 `daysLeft` 用「本地零点」与「`expire_date` 按 UTC 解析」相减，跨时区边界可能**差 1 天**（提醒早/晚一天）。建议统一用日期字符串比较。
- `patentBatchQueryJob.generateSystemToken` 依赖 **ERP 与 IP 系统共享 `JWT_SECRET`** 自签 `role:admin` token（跨系统密钥强耦合，单点泄漏即双系统沦陷；属已知架构，建议改为独立服务凭证）。
- IP 扫描会覆盖 `next_fee_deadline`，而 `inventoryService._refreshMaintainAggregates` 用年费 `MAX(deadline_date)` 也写该字段，两者可能**互相覆盖**（轻微）。

### 异常检测（P2）
- `patentAnomalyService.detectAndRecord` 仅检测**近 30 天**事件。新入库专利在 30 天前发生的转让/质押**不会被回溯告警**（漏报历史风险）。建议：专利首次入库或首次扫描时做一次"全量基线"检测。
- `feesDue` 中的"著录项目变更费"未做时间过滤（与 `feesPaid` 不一致），一般无害（应缴即当前），仅记录。
