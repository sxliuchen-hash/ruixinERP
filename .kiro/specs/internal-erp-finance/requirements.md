# Requirements: 内部财务管理系统 (Internal ERP Finance)

## Overview

为10人专利买卖公司开发内部财务管理系统，与企业微信审批打通，实时展示公司账目、现金流、利润、成本情况。管理囤积专利库存。复用主项目 `patent-notice-system` 用户体系，部署于 `erp.iptt.top`。

### 业务背景

- **代理交易**：客户选定专利 → 签销售合同收款 → 签采购合同向供应商付款
- **收购囤积**：收购闲置专利 → 囤积（持续年费） → 加价卖出
- 月均审批单据 50-100 单（含付款/收款/合同/发票/报销/借款/采购）
- 3 个公户 + 2 个私户，需分账户管理

## 一、数据接入

### Requirement 1: 企业微信审批同步

**User Story**
作为财务人员，我希望企业微信审批通过后数据自动进入系统，这样我不需要手动录入每一笔审批单据。

**Acceptance Criteria**
- [ ] 系统配置企业微信回调 URL，接收审批状态变更事件（`sys_approval_change`）
- [ ] 审批通过后自动解析审批详情并写入对应业务表（合同/收付款/发票/报销/借款）
- [ ] 同步进来的数据初始状态为"待确认"，财务确认并补充关联关系后变为"已确认"
- [ ] 定时任务每小时拉取最新审批数据作为兜底，防止回调丢失
- [ ] 使用审批单号 `sp_no` 做幂等去重，避免重复写入
- [ ] 支持 AES-256-CBC 消息加解密（回调验证 + 消息解析）
- [ ] 同步记录存入 `wechat_approvals` 表，含状态（待同步/已同步/同步失败/已忽略）
- [ ] 同步失败时记录错误原因，支持手动重试
- [ ] "待确认"列表在 Dashboard 醒目展示，提醒财务及时处理

### Requirement 2: 审批模板字段映射配置

**User Story**
作为管理员，我希望能在后台配置企业微信审批模板字段与系统字段的映射关系，这样审批模板调整时不需要改代码。

**Acceptance Criteria**
- [ ] 提供 `wechat_template_mappings` 配置表，存储模板 ID 与系统字段的映射
- [ ] 管理后台可查看、编辑映射关系
- [ ] 支持的映射目标：合同、收款、付款、发票、报销、借款
- [ ] 新增/修改映射后立即生效，无需重启服务
- [ ] 映射配置包含：模板 ID、模板字段 key、系统表名、系统字段名、数据类型转换规则

## 二、基础数据

### Requirement 3: 多账户管理

**User Story**
作为财务人员，我希望系统能区分公司的多个银行账户（3 个公户 + 2 个私户），这样每笔收付款都能对应到具体账户。

**Acceptance Criteria**
- [ ] 支持管理多个银行账户：账户名称、开户行、账号、账户类型（公户/私户）、状态
- [ ] 每个账户支持设置"期初余额"（系统上线时的实际余额，由管理员手动设置）
- [ ] 账户当前余额 = 期初余额 + 系统内收入 - 系统内支出，自动计算
- [ ] 每笔收款/付款必须选择对应的收付款账户
- [ ] Dashboard 可按账户维度查看余额和流水
- [ ] 支持账户间转账记录（如公户转私户），转账不影响公司总资产
- [ ] 定期通过银行流水对账校准余额（见 Requirement 12）

### Requirement 17: 客户与供应商管理

**User Story**
作为财务/业务员，我希望系统能管理客户和供应商信息，查看每个客户/供应商的往来账。

**Acceptance Criteria**
- [ ] 客户信息：名称、联系人、电话、地址、开票信息
- [ ] 供应商信息：名称、联系人、电话、地址、银行信息、税点
- [ ] 往来账：每个客户/供应商的历史交易汇总、应收/应付余额
- [ ] 与主项目客户数据通过 `client_relations` 表关联
- [ ] 支持按客户/供应商筛选合同和收付款记录

## 三、核心业务

### Requirement 4: 合同管理

**User Story**
作为业务员/财务，我希望系统能管理所有销售合同和采购合同，跟踪执行进度和关联的收付款情况。

**Acceptance Criteria**
- [ ] 支持销售合同和采购合同分类管理
- [ ] 合同信息包含：编号、类型、客户/供应商、金额、签订日期、状态（待签/执行中/已完成/已终止）
- [ ] 一个合同可关联多张发票、多次收付款
- [ ] 合同执行进度：已收/已付金额 vs 合同总额，自动计算完成百分比
- [ ] 支持上传合同扫描件附件（存储到腾讯云 COS，路径与主项目分离）
- [ ] 合同可关联专利（通过交易项目关联）
- [ ] 合同到期提醒（提前 30 天）
- [ ] 业务员只能看到自己相关的合同，财务和老板可看全部

### Requirement 5: 收付款管理

**User Story**
作为财务人员，我希望系统能记录每一笔收款和付款，关联到合同和账户，这样我能清楚知道每笔钱的来龙去脉。

**Acceptance Criteria**
- [ ] 收付款分为两种模式：
  - **业务类**（关联合同）：与销售/采购合同关联，用于交易项目利润核算
  - **费用类**（关联成本类别）：报销、网购、年费、日常支出等，不关联合同
- [ ] 收款记录：客户、金额、日期、收款方式（转账/支票/现金）、收款账户、关联合同（业务类）
- [ ] 付款记录：供应商、金额、日期、付款方式、付款账户、关联合同（业务类）或成本类别（费用类）
- [ ] 支持一个合同多次收付款（部分收付款）
- [ ] 应收管理：自动计算未回款金额（合同金额 - 已收金额）
- [ ] 应付管理：自动计算未付款金额（合同金额 - 已付金额）
- [ ] 企业微信审批通过后自动生成收付款记录，状态为"待确认"
- [ ] 支持手动录入收付款（非审批来源的）
- [ ] 业务类收付款关联到交易项目，用于利润核算
- [ ] 费用类支出自动归入成本管理模块

### Requirement 6: 发票管理

**User Story**
作为财务人员，我希望系统能管理开票和收票记录，并记录供应商的税点信息。

**Acceptance Criteria**
- [ ] 销项发票（开票）：关联销售合同、客户、金额、税额、发票号、开票日期
- [ ] 进项发票（收票）：关联采购合同、供应商、金额、税额、发票号
- [ ] 发票类型：普票/专票
- [ ] 开票状态：待开票/已开票/已作废
- [ ] 记录供应商税点（如 3 个点、6 个点），自动计算含税/不含税金额
- [ ] 发票与收付款的对应关系可查看

### Requirement 7: 报销与借款

**User Story**
作为员工，我希望报销和借款审批通过后自动进入系统；作为财务，我希望小额日常支出能自动归类。

**Acceptance Criteria**
- [ ] 报销记录：报销人、金额、费用类别、日期、备注、关联审批单号
- [ ] 借款记录：借款人、金额、日期、用途、还款状态（未还/部分归还/已还清）
- [ ] 还款记录：关联借款单，支持分次还款，自动更新还款状态
- [ ] 费用类别可自定义（交通/餐饮/办公/网购/物业/水电/其他）
- [ ] 自动识别归类：根据摘要关键词自动匹配费用类别
- [ ] 归类规则可配置：管理员可添加关键词→类别的映射规则
- [ ] 费用统计：按类别、按人、按月汇总
- [ ] 报销/借款关联到付款账户

### Requirement 8: 专利库存管理（囤积专利）

**User Story**
作为老板，我希望能看到所有囤积专利的库存情况、维持成本和库龄，方便我决定哪些专利需要调价或放弃。

**Acceptance Criteria**
- [ ] 专利独立入库：手动录入专利号、名称、类型、技术领域、采购价格、采购日期、供应商
- [ ] 入库时机：采购付款确认后，系统提示"是否将相关专利入库？"，由人工确认
- [ ] 库存状态管理：在库/已售出/已放弃/待转让
- [ ] 库龄自动计算：入库日期到当前的天数，支持按库龄排序和筛选
- [ ] 维持成本记录：年费、代理费等，每笔单独记录，累计自动汇总
- [ ] 定价管理：当前售价、历史调价记录（含调价日期、原因）
- [ ] 利润预估：预期售价 - 采购价 - 累计维持成本 = 预估利润
- [ ] 批量操作：批量调价、批量标记状态
- [ ] 年费到期提醒：提前 60 天、30 天、7 天提醒
- [ ] 库存总览：在库数量、总库存价值、平均库龄、总维持成本
- [ ] 支持按技术领域、库龄、利润率等维度筛选

## 四、分析决策

### Requirement 10: 交易项目管理

**User Story**
作为老板，我希望能按"单笔交易"维度看利润，知道每个专利交易赚了多少钱。

**Acceptance Criteria**
- [ ] 交易项目包含：项目名称、关联专利、客户、供应商、状态（进行中/已完成/已取消）
- [ ] 一个交易项目可关联：1 个销售合同 + 1 个采购合同 + 多次收付款 + 维持成本
- [ ] 自动计算单笔**毛利润**：销售收入 - 采购成本 - 供应商税点 - 直接维持成本
- [ ] 交易项目列表：按利润、状态、日期排序筛选
- [ ] 交易项目详情页：展示完整的资金流向（收了多少、付了多少、成本多少、利润多少）
- [ ] 支持囤积专利的交易：采购入库 → 持续维持 → 最终售出，全生命周期利润核算
- [ ] Dashboard 展示两个利润口径：
  - **毛利润**（交易层面）= 所有已完成交易的毛利润之和
  - **净利润**（公司层面）= 毛利润合计 - 人力成本 - 运营成本 - 其他固定费用

### Requirement 11: 成本管理

**User Story**
作为老板，我希望能清楚看到公司各项成本的构成和趋势，帮助我控制开支。

**Acceptance Criteria**
- [ ] 成本分类：人力成本（工资/社保/公积金）、运营成本（房租/水电/网络/物业/办公）、专利维持成本（年费/代理费）、营销成本
- [ ] 成本分类可自定义扩展（管理员可新增类别）
- [ ] 人力成本按月按人录入
- [ ] 运营成本按月录入，支持设置固定月费（如房租每月自动生成）
- [ ] 专利维持成本从专利库存模块自动汇总
- [ ] 按月/季/年维度汇总，支持同比环比
- [ ] 成本占比饼图、月度趋势折线图

### Requirement 13: Dashboard 看板

**User Story**
作为老板，我希望打开系统就能一眼看到公司的财务全貌，包括现金流、利润、成本、库存等核心指标。

**Acceptance Criteria**
- [ ] 打开即刷新，实时计算展示数据
- [ ] 核心指标卡片：现金流入、现金流出、净现金流、毛利润、净利润、应收账款、应付账款
- [ ] 利润双口径展示：
  - 毛利润 = 已完成交易的（销售收入 - 采购成本 - 税点 - 直接维持成本）之和
  - 净利润 = 毛利润 - 人力成本 - 运营成本 - 其他固定费用
- [ ] 按账户展示余额（3 公户 + 2 私户分别显示，含期初余额说明）
- [ ] 专利库存概览：在库数量、总价值、平均库龄
- [ ] 成本构成饼图：人力/运营/专利维持/采购占比
- [ ] 月度趋势折线图：收入/支出/毛利润/净利润近 12 个月趋势
- [ ] 时间范围切换：本月/本季/本年/自定义
- [ ] 本月新签合同数量和金额
- [ ] 应收账龄分布（30 天内 / 30-60 天 / 60-90 天 / 90 天以上）
- [ ] "待确认"单据数量提醒（企业微信同步进来但未确认的）

## 五、提醒通知

### Requirement 9: 年费到期提醒（企业微信推送）

**User Story**
作为老板/财务，我希望专利年费快到期时能收到企业微信消息提醒，避免因忘缴年费导致专利失效。

**Acceptance Criteria**
- [ ] 定时任务每日检查所有在库专利的年费到期日
- [ ] 提前 60 天、30 天、7 天分别推送提醒
- [ ] 通过企业微信应用消息推送给指定负责人
- [ ] 提醒内容包含：专利号、专利名称、到期日期、距到期天数
- [ ] 已缴费的专利不再重复提醒
- [ ] 合同到期也通过同样机制提醒（提前 30 天）

## 六、对账核验

### Requirement 12: 银行流水对账

**User Story**
作为财务人员，我希望能导入银行公户的流水 Excel，与系统记录自动对比，快速发现漏记的账目。

**Acceptance Criteria**
- [ ] 支持导入银行流水 Excel 文件（支持常见银行导出格式）
- [ ] 自动匹配：根据金额、日期、摘要与系统收付款记录进行匹配
- [ ] 对账结果展示：已匹配 / 未匹配（可能漏记）/ 系统多余（系统有但流水无）
- [ ] 未匹配的流水可一键创建收付款记录
- [ ] 对账历史记录保存，可追溯
- [ ] 支持按账户选择对账（3 个公户分别对账）

## 七、工具模块

### Requirement 14: 数据导出

**User Story**
作为财务人员，我希望能导出收支明细 Excel 给外包会计报税使用，也能导出各类报表供老板查看。

**Acceptance Criteria**
- [ ] 支持导出收支明细 Excel（按月/按季/按年/自定义时间范围）
- [ ] 支持导出合同列表、发票列表、报销明细
- [ ] 支持导出专利库存清单（含库龄、维持成本、利润预估）
- [ ] 导出格式为 .xlsx，含表头和格式化
- [ ] 导出时可选择筛选条件（账户、类别、状态等）

### Requirement 15: 历史数据迁移

**User Story**
作为管理员，我希望能将历史 Excel 数据批量导入系统，这样系统上线后 Dashboard 就有完整的历史数据参考。

**Acceptance Criteria**
- [ ] 提供 Excel 导入模板下载（合同、收付款、专利库存、成本等）
- [ ] 支持批量导入：合同（约 2000 条）、收付款记录、专利库存、成本记录
- [ ] 导入前数据校验：必填字段检查、格式检查、重复检查
- [ ] 导入结果报告：成功数、失败数、失败原因明细
- [ ] 支持增量导入（不覆盖已有数据）
- [ ] 导入过程支持回滚（导入失败时不影响已有数据）

### Requirement 16: 自动识别归类

**User Story**
作为财务人员，我希望系统能根据摘要关键词自动识别日常支出的类别，减少我手动分类的工作量。

**Acceptance Criteria**
- [ ] 系统预置常见关键词→类别映射（如"物业费"→运营-物业、"水电"→运营-水电）
- [ ] 管理员可自定义添加/修改/删除关键词映射规则
- [ ] 银行流水导入时，根据摘要自动建议类别（重点场景，银行摘要格式规范）
- [ ] 企业微信审批同步时，根据摘要尝试匹配类别（辅助场景）
- [ ] 自动归类结果标记为"建议"，需人工确认后生效
- [ ] 初期从简：纯关键词匹配，不做复杂 NLP；后续根据使用情况优化规则

## 八、系统基础

### Requirement 18: 角色权限管理

**User Story**
作为管理员，我希望不同角色只能看到和操作自己权限范围内的数据。

**Acceptance Criteria**
- [ ] 复用主项目 users 表，通过 role 字段判断权限
- [ ] admin（老板）：全部数据查看、系统配置、所有操作
- [ ] process（财务）：录入/编辑账目、对账、查看所有财务数据、导出
- [ ] agent（业务员）：查看自己相关的合同和收付款、提交报销/借款、不可看他人数据
- [ ] client/sub_account/sub_department：不可访问 ERP 模块
- [ ] 敏感操作（删除、金额修改）需二次确认
- [ ] 所有数据变更记录操作日志

### Requirement 19: 系统切换与导航

**User Story**
作为用户，我希望能在官文管理系统和 ERP 系统之间无缝切换，不需要重新登录。

**Acceptance Criteria**
- [ ] 顶部导航栏增加系统切换入口（官文管理 ↔ ERP 财务）
- [ ] 共享 JWT Token，切换时无需重新登录
- [ ] 主项目也增加跳转到 ERP 的入口

## 十一、关键设计决策

| # | 决策 | 原因 |
|---|------|------|
| 1 | 审批同步后数据为"待确认"状态 | 审批信息可能不完整，需财务补充关联关系 |
| 2 | 收付款区分"业务类"和"费用类" | 不是所有支出都有合同（报销/网购/年费无合同） |
| 3 | 利润分"毛利润"和"净利润"两个口径 | 老板既要看业务赚多少，也要看扣除固定开支后实际赚多少 |
| 4 | 专利入库为手动确认操作 | 付款 ≠ 权利转移，需人工确认后再入库 |
| 5 | 账户设置"期初余额" | 解决系统上线时历史数据不完整导致余额不准的问题 |
| 6 | 自动归类初期从简（关键词匹配） | 实际需归类的小额支出量不大，过度设计性价比低 |
